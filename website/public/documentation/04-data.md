# 04 - Data and Resources

## 1) Where data is stored

All persistent data is located in the `django_proxy/data/` directory:

```text
django_proxy/data/
├── areas.gpkg                    # Geographic areas (regions, zones)
├── deals.gpkg                    # Land investment deals
├── world_region_light.gpkg       # World regions reference layer
├── reports/
│   └── report_*.json             # Generated analysis reports
└── *.geojson                     # Test or supplementary data
```

---

## 2) Data formats

### GeoPackage (.gpkg)
GeoPackage is an open standard for geospatial vector data. It stores:
- geometries (points, lines, polygons),
- attributes (tabular data linked to geometries),
- spatial indexes,
- metadata.

**Main files**:
- **areas.gpkg**: contains area-based land investment data.
- **deals.gpkg**: contains point or polygon locations of individual deals.
- **world_region_light.gpkg**: reference layer for global context.

### GeoJSON
A JSON-based geospatial format, human-readable and widely supported.
- Used for test data or interchange with other tools.

### JSON reports
Generated by analysis workflows and stored in `reports/`:
- naming pattern: `report_YYYY_MM_DD_HH_MM_SS.json`
- contain statistics, summaries, and metadata.

## 3) Data structure (critical schema)

The project relies on a specific JSON structure provided by the Land Matrix API. Any modification to the schema can cause failures.

### Root-level fields
- **`id`**: unique identifier for each record.
- **`country_id`**: country code for filtering and mapping.

### Version-specific fields (inside `selected_version`)
- **`deal_size`**: area in hectares.
- **`current_intention_of_investment`**: purpose category (agriculture, mining, etc.).
- **`current_implementation_status`**: implementation phase.
- **`current_negotiation_status`**: contractual status.
- **`initiation_year`**: year the deal started.

### Geographic data
- **`locations`**: must be an iterable list inside `selected_version`.
- Each location object must include:
  - **`level_of_accuracy`**: precision level for mapping. Valid values:
    - `EXACT_LOCATION`: precise coordinates
    - `APPROXIMATE_LOCATION`: approximate location
    - `COORDINATES`: general coordinates
    - `COUNTRY`: country-level only (no specific location)
  - coordinates or geometry information.

---

## 4) Why this structure matters

The application code depends on these fields to:
- display deals on the map,
- filter by criteria (country, year, investment type),
- compute statistics and generate reports,
- avoid runtime errors due to missing keys.

**Do not rename, remove, or restructure these fields** unless you are prepared to update all dependent code.

---

## 5) Data import and export

### Import
Data is imported through the crawler module:
- source: Land Matrix API (JSON format).
- destination: local GeoPackage files.

### Export
Users can export processed data in multiple formats:
- **PDF**: reports with maps and tables.
- **Excel**: tabular data with formatting.
- **CSV**: plain tabular export.
- **GeoJSON**: spatial export for GIS tools.

---

## 6) Test data

- **Test GeoJSON files** are sometimes included for validating spatial queries.
- Test polygons help verify intersection logic and filtering behavior.

---

## 7) Summary

Data in this project is stored primarily in GeoPackage files, with a strict schema enforced by the Land Matrix API. Maintaining this schema ensures all components work correctly. Reports and logs provide traceability for analysis workflows.
